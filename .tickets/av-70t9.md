# Shim fallback for File System Access pickers (showOpenFilePicker) in sandboxed artifacts

Artifacts that call the File System Access API (showOpenFilePicker / showSaveFilePicker / showDirectoryPicker) fail in our sandboxed iframe with the Chromium error: "Sandboxed documents aren't allowed to show a file picker."

## Root cause (verified in Chromium source)

The error is thrown by Blink's VerifyIsAllowedToShowFilePicker in third_party/blink/renderer/modules/file_system_access/global_file_system_access.cc. The check ladder:

1. Opaque origin (sandbox without allow-same-origin) -> SecurityError "Sandboxed documents aren't allowed to show a file picker." (what artifacts hit)
2. Even with a real origin, a frame cross-origin to the outermost main frame -> "Cross origin sub frames aren't allowed to show a file picker."
3. Also requires transient user activation.

Consequences:
- NO sandbox token re-enables the FSA API. Even allow-same-origin (forbidden by our security model, docs/architecture.md section 4) would not help: RENDER_ORIGIN is cross-origin to APP_ORIGIN, so check 2 still fires. FSA is unreachable inside our iframe by design.
- The classic <input type="file"> picker is NOT blocked: Blink's file_input_type.cc has no sandbox check at all, only a user-activation requirement. File drag-and-drop into the iframe also works. So only artifacts written against showOpenFilePicker are affected.

## Proposed fix: polyfill in the storage shim

Extend the injected shim (already the first <head> script in every rendered artifact) to install showOpenFilePicker / showDirectoryPicker / showSaveFilePicker replacements:
- Open: hidden <input type=file> (webkitdirectory for directories), return FSA-shaped handle objects whose getFile() resolves to the picked File. Same fallback ladder as Google's browser-fs-access library (github.com/GoogleChromeLabs/browser-fs-access).
- Save: createWritable fallback that materializes a download. Requires adding the standard 'allow-downloads' token to the iframe sandbox attribute (does not weaken origin isolation).

Fits 'observe, don't predict': unconditional install, idle at zero cost if unused. Bonus: makes FSA-written artifacts work in Firefox/Safari, which don't ship the API at all.

No permission gating needed for the input fallback - the user explicitly picks each file (ordinary web behavior). If desired, 'allow-downloads' could be a per-artifact toggle alongside the network allowlist, but it is low-risk enough to grant globally.

## Rejected alternative

Host-frame picker bridge (shim postMessage -> host calls real showOpenFilePicker; legal because user activation propagates from iframe click to ancestor frames; host posts File/Blob contents back, never the FileSystemHandle). Real machinery, only worth it if artifacts ever need writable disk handles. Tier-1/2 tools only need read-a-picked-file and export-a-result, which the shim polyfill covers.

## Acceptance Criteria

- Artifact calling showOpenFilePicker inside the sandboxed iframe opens a working picker and receives handle-like objects; getFile() resolves to the picked File
- showDirectoryPicker works via webkitdirectory fallback
- showSaveFilePicker createWritable fallback produces a download; iframe sandbox gains allow-downloads
- Shim installs unconditionally before artifact scripts; no behavior change for artifacts that never call FSA
- Plain <input type=file> artifacts continue to work unchanged

## Notes

**2026-07-12T20:40:00Z**

Implemented in the render shim (`internal/render/render.go`) as the fourth bridge alongside storage/download/clipboard. All three pickers are polyfilled on `<input type=file>` (the one picker Blink allows inside a sandboxed frame — no sandbox check in `file_input_type.cc`, only user activation, which the artifact's own click already provides):

- `showOpenFilePicker` → hidden `<input type=file multiple?>`, returns FSA-shaped file handles (`getFile()` resolves to the picked `File`). `options.types` maps to the `accept` attribute.
- `showDirectoryPicker` → `<input type=file webkitdirectory>`; rebuilds the directory tree from each `File.webkitRelativePath` so `dir.values()` / `entries()` / `keys()` and the async iterator yield real nested handles (not a flat list). Empty subdirs are invisible to `webkitdirectory` and are omitted.
- `showSaveFilePicker` → returns a handle whose `createWritable()` buffers writes and, on `close()`, materializes a download. `write()` accepts both bare `Blob`/`ArrayBuffer`/`string` and the WriteParams form `{type:'write',data}`; `seek`/`truncate` are no-ops (sequential buffer). Canceled pickers reject with `AbortError`, matching native FSA.

**Deviation from the original plan (intentional, stronger):** the ticket proposed adding `allow-downloads` to the iframe sandbox for the save fallback. That was written before av-ryby landed its host-mediated download bridge. The save path instead reuses that bridge — `close()` creates a detached blob:-href anchor and clicks it, which the existing download-interception routes to the host with first-use approval. The sandbox token set stays exactly `allow-scripts` (the `downloads_test.go` invariant holds), and there is no new host message type (saves ride `__avDownload` like every other export). Single export path, no weakening.

Framed-only (inside the `if (window.parent !== window)` guard, same as download/clipboard): top-level renders have native FSA, share pages get no bridge. Unconditional install — idle at zero cost if unused. Tests in `internal/render/render_test.go` cover the API surface, the input/`webkitdirectory`/`webkitRelativePath` fallbacks, `AbortError` on cancel, async-iterable directory handles, and the save-routes-through-download-bridge invariant.
